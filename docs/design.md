# DataWorks 加维度 Web 工具设计文档

## 目标
将 `dataworks-add-dimension` skill 转化为可视化 Web 工具，支持用户在页面上完成：选择项目空间 → 输入表名和维度 → 分析上下游 → 修改 SQL → 保存草稿 → 可选执行 alter table → 可选历史回刷 → 可选 BI 平台同步。

## 技术栈
- **前端**：Vue 3 + Vite + Element Plus
- **后端**：Python FastAPI + uvicorn
- **部署**：本机运行，前后端分离

### 5 步流程映射

| 步骤 | 页面标题 | 关键操作 |
|------|---------|---------|
| 1. 配置 | 选择项目 & 输入维度 | 下拉选择项目空间、输入表名、维度列名 |
| 2. 分析 | 上下游分析结果 | 展示上游表字段存在性、下游节点列表 |
| 3. 修改 | SQL 修改预览 | 展示 diff、修改 SQL、自动补全 cube/group by |
| 4. 执行 | 执行确认 | alter table（可选）、历史回刷（可选） |
| 5. 完成 | 结果 & BI 同步 | 展示修改结果、可选同步 BI 筛选器 |

### 关键交互设计

**Step 1 - 配置：**
- 项目空间：下拉选择（从 DataWorks 拉取）
- 表名：文本输入，支持模糊搜索已有表
- 新增维度：文本输入 + 中文名称
- "开始分析"按钮

**Step 2 - 分析：**
- 上游分析：表格展示各上游表是否包含该维度字段
  - ✅ 存在 → 显示来源表名
  - ❌ 不存在 → 标红，提示"需从上游补充"
- 下游分析：复选框列表，用户勾选需要修改的下游节点
- 维度展开方式：自动识别（cube / LATERAL VIEW / group by）
- 问题检测：如果有上游缺失，弹出选择框让用户决定
  - 选项1：从某个上游表补充
  - 选项2：使用 LEFT JOIN 关联维度表
  - 选项3：停止，手动处理

**Step 3 - 修改：**
- 左侧：原始 SQL（只读）
- 右侧：修改后 SQL（可编辑，高亮新增行）
- diff 预览：绿色表示新增，红色表示删除
- 自动检查点：
  - 新增列在目标 schema 最后一个 column
  - ALL 口径写法正确
  - alter table SQL 已生成

**Step 4 - 执行（汇总确认）：**
- 最终确认清单（Checklist）：
  - [x] SQL 修改正确
  - [ ] 执行 alter table（可选，默认不勾选）
  - [ ] 历史回刷（可选，默认不勾选）
    - 如果勾选，展示回刷 SQL 和影响分区数估算
  - [ ] 提交到 ODPS（可选）
  - [ ] 同步 BI 筛选器（可选）
- "确认执行"按钮

**Step 5 - 完成：**
- 执行结果展示
- 修改的节点信息（project_id, file_id, node_id）
- BI 同步结果（如果执行了）
- "重新开始"按钮

## 后端 API 设计

### API 列表

```
GET  /api/projects                   → 获取项目空间列表
GET  /api/tables?project={id}        → 获取项目下表列表（模糊搜索）
POST /api/analyze                    → 提交表名+维度，返回上下游分析
POST /api/modify                     → 提交修改参数，返回修改后的 SQL + diff
POST /api/save-draft                 → 保存到 DataWorks 草稿
POST /api/execute-alter              → 执行 alter table
POST /api/generate-backfill          → 生成历史回刷 SQL
POST /api/execute-backfill           → 执行历史回刷
POST /api/sync-bi                    → 同步 BI 平台筛选器
```

### 复用现有脚本

后端通过 `subprocess` 调用现有 Python 脚本：
- `scripts/query_dataworks_task.py` → 获取 SQL
- `scripts/find_downstream_tasks.py` → 获取下游
- `scripts/run_odps_sql.py` → 执行 SQL
- `scripts/commit_dataworks_node.py` → 保存草稿
- `scripts/manage_sql_blocks.py` → 注释 alter 块

## 数据流

```
用户选择项目 + 输入表名 + 输入维度
  ↓
后端调用 query_dataworks_task.py 获取 SQL
  ↓
解析 SQL → 识别上游表 → 分析字段存在性
  ↓
如果有上游缺失 → 返回问题让用户选择
  ↓
用户确认后 → 修改 SQL（补维度、补 cube/group by）
  ↓
diff 预览 → 用户确认
  ↓
保存草稿 → 可选 alter table → 可选历史回刷
  ↓
可选 BI 平台同步
  ↓
完成
```

## 确认机制

- **关键节点强制确认**（弹窗）：
  - alter table 执行前
  - 历史回刷执行前
  - 提交到 ODPS 前

- **次要节点汇总确认**（最终 checklist）：
  - SQL diff 确认
  - 下游节点修改范围
  - BI 同步范围

## 已知缺陷 & 改进

1. **上游校验**：原 skill 仅"询问用户"，改进为表格化展示 + 下拉选择来源
2. **下游修改**：原 skill 自动处理所有下游，改进为复选框让用户选择
3. **diff 预览**：原 skill 无此环节，新增可视化 diff
4. **回滚机制**：原 skill 无此环节，新增草稿备份
5. **历史回刷分区估算**：原 skill 无此环节，新增分区范围计算
6. **自检结果展示**：原 skill 后台运行，改进为页面展示
7. **环境预检查**：新增依赖检测页面
